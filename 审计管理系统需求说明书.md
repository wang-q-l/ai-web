## 二、项目概述

### 2.1 项目背景

随着企业规模的不断扩大和业务复杂度的提升，审计工作在企业管理中扮演着越来越重要的角色。传统的审计管理方式存在以下问题：

- **流程管理不规范**：审计项目从准备、实施到报告的各个阶段缺乏统一的管理平台，导致流程不清晰、责任不明确
- **整改跟踪困难**：审计发现的问题整改情况难以实时跟踪，整改效果难以评估，容易出现问题反复
- **协同效率低下**：审计人员与迎审人员之间的沟通协作主要依赖线下方式，信息传递效率低、易出错
- **数据分散管理**：审计资料、整改方案、整改进展等数据分散在各个部门，缺乏统一的数据管理和查询平台
- **角色权限混乱**：不同角色的人员权限管理不清晰，容易出现越权操作或权限不足的情况

为解决上述问题，提升审计管理的规范化、信息化水平，特建设审计管理系统，实现审计项目全流程管理和整改问题闭环管理。

### 2.2 项目建设目标

本项目旨在建设一套完整的审计管理系统，实现以下核心目标：

1. **规范审计流程管理**：建立标准化的审计项目管理流程，覆盖准备阶段、实施阶段、报告阶段、后续审计四大阶段，确保审计工作有序开展
2. **实现整改闭环管理**：建立整改问题全生命周期管理机制，从问题下发、方案反馈、进展跟踪到整改报告，实现整改工作的闭环管理
3. **提升协同工作效率**：通过系统化的角色权限管理和消息通知机制，实现审计人员与迎审人员之间的高效协作
4. **统一数据管理平台**：建立统一的数据管理和查询平台，实现审计资料、整改方案、整改进展等数据的集中管理和快速查询
5. **强化数据分析能力**：通过报表查询和数据统计功能，为管理层提供决策支持，提升审计管理的科学性和有效性

### 2.3 项目建设范围

本项目建设范围包括以下内容：

**包含的功能模块：**

- **项目管理子系统**：人员管理、项目信息查看、启动项目、项目作业（准备阶段、实施阶段、报告阶段、后续审计）
- **整改管理子系统**：整改项目维护、整改问题维护、整改问题下发、整改方案反馈、整改进展反馈、整改报告、整改台账、报表查询

**涉及的端：**

- **Web 管理后台**：审计人员和迎审人员使用的主要工作平台，支持 PC 端浏览器访问

**不包含的功能：**

- 移动端 App 开发（本期不涉及）
- 与外部审计系统的数据对接（本期不涉及）
- 审计知识库管理（本期不涉及）

### 2.4 业务流程图

审计管理系统的业务流程涉及审计人员和迎审人员两类角色的协作，通过泳道图展示各角色在审计项目实施和整改管理过程中的职责和交互关系。

![审计业务流程图](images/audit-business-flow.png)

**流程说明：**

1. **审计项目启动阶段**：审计人员创建审计项目，设置项目基本信息和参与人员
2. **审计作业执行阶段**：审计人员执行审计作业，包括准备阶段、实施阶段、报告阶段三个环节
3. **问题下发阶段**：审计人员发现问题后，将整改问题下发给迎审人员（接收人员）
4. **方案反馈阶段**：迎审人员（接收人员）接收整改问题，制定并提交整改方案
5. **整改执行阶段**：迎审人员（经办人员）根据整改方案执行具体整改任务
6. **进展反馈阶段**：迎审人员（经办人员）定期反馈整改进展，上传整改证明材料
7. **跟踪验收阶段**：审计人员跟踪整改进展，对整改结果进行验收
8. **报告生成阶段**：系统生成整改报告，汇总整改情况和效果评估

### 2.5 用户角色定义

系统设有以下用户角色：

| 角色名称 | 角色描述 | 主要职责 | 权限范围 |
|---------|---------|---------|---------|
| 审计人员 | 负责审计项目的执行和管理 | 创建审计项目、执行审计作业、下发整改问题、跟踪整改进展 | 项目管理、整改管理的全部功能 |
| 迎审人员（接收人员） | 接收审计问题和整改任务 | 接收整改问题、查看整改要求、提交整改方案 | 查看整改问题、提交整改方案 |
| 迎审人员（经办人员） | 具体执行整改工作 | 执行整改任务、反馈整改进展、提交整改报告 | 查看整改问题、反馈整改进展、提交整改报告 |
| 系统管理员 | 系统维护和用户管理 | 用户管理、权限配置、系统参数设置 | 系统管理的全部功能 |

**角色切换机制：**

- 系统支持顶部导航切换角色，用户可以在审计人员和迎审人员角色之间切换
- 切换角色后，系统界面和功能菜单会根据当前角色的权限进行调整

### 3.3 功能详细设计

#### 3.3.1 人员管理

##### （一）功能需求概述

人员管理模块为审计人员提供审计项目参与人员的管理功能，包括审计人员和迎审人员的信息维护、角色分配、权限设置等操作。通过该模块，可以统一管理项目相关人员，确保审计工作的人员配置合理、职责明确。

##### （二）用户操作流程

**添加人员流程：**

1. 审计人员进入人员管理页面
2. 点击【添加人员】按钮
3. 系统打开添加人员弹窗
4. 审计人员填写人员信息：
   - 姓名（必填）
   - 工号（必填）
   - 所属部门（必填）
   - 联系电话（必填）
   - 邮箱（必填）
   - 角色类型（必填）：审计人员/迎审人员（接收）/迎审人员（经办）
5. 审计人员点击【保存】按钮
6. 系统验证数据：
   - 验证通过：保存人员信息，显示成功提示，关闭弹窗，刷新列表
   - 验证失败：显示错误提示，停留在弹窗

**编辑人员流程：**

1. 审计人员在人员列表中找到目标人员
2. 点击操作列的【编辑】按钮
3. 系统打开编辑弹窗，回显人员信息
4. 审计人员修改人员信息
5. 点击【保存】按钮
6. 系统验证并保存，刷新列表

**删除人员流程：**

1. 审计人员在人员列表中找到目标人员
2. 点击操作列的【删除】按钮
3. 系统弹出二次确认对话框："确定要删除该人员吗？"
4. 审计人员点击【确定】
5. 系统检查人员是否被项目引用：
   - 未被引用：执行删除，显示成功提示，刷新列表
   - 已被引用：显示错误提示"该人员已被项目引用，无法删除"

##### （三）功能设计

**核心功能：**

- 人员列表展示：分页展示人员信息，支持按角色类型、部门筛选
- 多条件筛选：支持按姓名、工号、角色类型、部门筛选
- 添加人员：打开添加弹窗，填写人员信息
- 编辑人员：打开编辑弹窗，回显并修改人员信息
- 删除人员：删除人员信息，需二次确认
- 批量删除：选中多个人员批量删除
- 导出人员：支持导出为 Excel 文件

**数据字段：**

| 字段名称 | 字段类型 | 长度限制 | 是否必填 | 默认值 | 验证规则 | 说明 |
|---------|---------|---------|---------|--------|---------|------|
| 人员ID | number | - | 自动生成 | - | - | 唯一标识 |
| 姓名 | string | 50 | 是 | - | 非空 | 人员姓名 |
| 工号 | string | 20 | 是 | - | 非空，唯一 | 员工工号 |
| 所属部门 | string | 100 | 是 | - | 非空 | 部门名称 |
| 联系电话 | string | 11 | 是 | - | 手机号格式 | 联系方式 |
| 邮箱 | string | 100 | 是 | - | 邮箱格式 | 电子邮箱 |
| 角色类型 | number | - | 是 | 1 | 1-3 | 1-审计人员，2-迎审人员（接收），3-迎审人员（经办） |
| 创建时间 | datetime | - | 自动生成 | - | - | 创建时间 |
| 更新时间 | datetime | - | 自动更新 | - | - | 最后更新时间 |
| 状态 | number | - | 是 | 1 | 0或1 | 1-启用，0-停用 |

**业务规则：**

- 工号必须唯一，不允许重复
- 联系电话必须符合手机号格式（11位数字）
- 邮箱必须符合邮箱格式
- 已被项目引用的人员不允许删除，只能停用
- 停用的人员不能被分配到新项目

##### （四）业务逻辑关联

- **项目引用关系**：人员被【启动项目】模块引用，用于分配项目参与人员
- **整改问题关联**：人员被【整改问题下发】模块引用，用于指定接收人员和经办人员
- **数据流向**：人员管理 → 启动项目 → 整改问题下发
- **删除限制**：如果人员已被项目引用，则不允许删除，只能停用；系统会检查人员的引用关系

#### 3.3.2 项目信息查看

##### （一）功能需求概述

项目信息查看模块为所有用户提供审计项目的基本信息查看功能，包括项目名称、项目编号、项目状态、参与人员、项目进度等信息。用户可以通过该模块了解项目的整体情况，为后续工作提供参考。

##### （二）用户操作流程

**查看项目列表流程：**

1. 用户进入项目信息查看页面
2. 系统展示项目列表，包括项目名称、项目编号、项目状态、开始时间、结束时间等
3. 用户可以按项目名称、项目状态、时间范围筛选项目
4. 用户点击【查看详情】按钮
5. 系统打开项目详情页面，展示项目完整信息

**查看项目详情流程：**

1. 用户在项目列表中点击【查看详情】按钮
2. 系统打开项目详情页面
3. 页面展示项目基本信息、参与人员、项目进度、审计作业记录等
4. 用户可以查看各阶段的工作内容和完成情况
5. 用户点击【返回】按钮返回项目列表

##### （三）功能设计

**核心功能：**

- 项目列表展示：分页展示项目信息，支持按状态、时间范围筛选
- 多条件筛选：支持按项目名称、项目编号、项目状态、时间范围筛选
- 查看项目详情：展示项目完整信息，包括基本信息、参与人员、项目进度
- 查看审计作业：展示各阶段的审计作业记录
- 查看整改问题：展示项目相关的整改问题列表
- 导出项目信息：支持导出为 Excel 或 PDF 文件

**数据字段：**

| 字段名称 | 字段类型 | 长度限制 | 是否必填 | 默认值 | 验证规则 | 说明 |
|---------|---------|---------|---------|--------|---------|------|
| 项目ID | number | - | 自动生成 | - | - | 唯一标识 |
| 项目名称 | string | 100 | 是 | - | 非空 | 项目名称 |
| 项目编号 | string | 50 | 是 | - | 非空，唯一 | 项目编号 |
| 项目类型 | number | - | 是 | 1 | 1-5 | 1-财务审计，2-合规审计，3-内控审计，4-专项审计，5-其他 |
| 项目状态 | number | - | 是 | 1 | 1-4 | 1-准备阶段，2-实施阶段，3-报告阶段，4-后续审计 |
| 审计对象 | string | 200 | 是 | - | 非空 | 被审计单位或部门 |
| 开始时间 | date | - | 是 | - | 非空 | 项目开始日期 |
| 结束时间 | date | - | 是 | - | 非空，大于开始时间 | 项目结束日期 |
| 项目负责人 | number | - | 是 | - | 非空 | 关联人员表 |
| 参与人员 | array | - | 是 | - | 非空 | 关联人员表，JSON格式 |
| 项目描述 | string | 500 | 否 | - | - | 项目说明 |
| 创建时间 | datetime | - | 自动生成 | - | - | 创建时间 |
| 更新时间 | datetime | - | 自动更新 | - | - | 最后更新时间 |

**业务规则：**

- 项目编号必须唯一，不允许重复
- 结束时间必须大于开始时间
- 项目负责人必须是审计人员角色
- 参与人员至少包含1名审计人员
- 项目状态按顺序流转：准备阶段 → 实施阶段 → 报告阶段 → 后续审计

##### （四）业务逻辑关联

## 四、技术方案

### 4.1 技术架构

#### 4.1.1 前端技术栈

- **框架**：Vue 3.5 + TypeScript 5.6
- **构建工具**：Vite 7.1
- **UI 组件库**：Element Plus 2.11
- **状态管理**：Pinia 3.0
- **路由管理**：Vue Router 4.x
- **HTTP 请求**：Axios
- **代码规范**：ESLint + Prettier + Stylelint
- **提交规范**：Husky + Commitizen

#### 4.1.2 后端技术栈

- **开发语言**：Java 17 / Node.js 18+
- **开发框架**：Spring Boot 3.x / Express.js
- **数据库**：MySQL 8.0 / PostgreSQL 14+
- **缓存**：Redis 7.x
- **文件存储**：本地存储 / 阿里云 OSS / MinIO
- **接口文档**：Swagger / Apifox

#### 4.1.3 系统架构设计

系统采用前后端分离架构，前端使用 Vue 3 构建单页应用（SPA），后端提供 RESTful API 接口。

**架构层次：**

- **表现层**：Vue 3 + Element Plus，负责用户界面展示和交互
- **业务逻辑层**：后端服务，负责业务逻辑处理和数据验证
- **数据访问层**：ORM 框架，负责数据库操作
- **数据存储层**：MySQL 数据库，负责数据持久化

### 4.2 数据交互

#### 4.2.1 前后端数据交互方式

- **协议**：HTTP/HTTPS
- **数据格式**：JSON
- **请求方式**：GET、POST、PUT、DELETE
- **认证方式**：JWT Token

#### 4.2.2 API 设计原则

- **RESTful 风格**：遵循 REST 架构风格，使用标准 HTTP 方法
- **统一响应格式**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {}
  }
  ```
- **错误处理**：统一的错误码和错误信息
- **分页规范**：统一的分页参数（page、pageSize）和响应格式
- **接口版本管理**：通过 URL 路径管理接口版本（如 /api/v1/）

#### 4.2.3 接口命名规范

- **资源命名**：使用复数名词（如 /users、/projects）
- **操作命名**：使用 HTTP 方法表示操作（GET 查询、POST 创建、PUT 更新、DELETE 删除）
- **路径参数**：使用 RESTful 风格（如 /users/:id）
- **查询参数**：使用 query string（如 ?name=xxx&status=1）

### 4.3 安全方案

#### 4.3.1 身份认证

- **登录认证**：用户名密码登录，支持验证码验证
- **Token 机制**：使用 JWT Token 进行身份认证
- **Token 刷新**：支持 Token 自动刷新机制
- **单点登录**：支持与企业 SSO 系统集成（可选）

#### 4.3.2 权限控制

- **角色权限**：基于角色的访问控制（RBAC）
- **菜单权限**：根据用户角色动态加载菜单
- **按钮权限**：根据用户角色控制页面按钮的显示和操作
- **数据权限**：根据用户角色控制数据的访问范围

#### 4.3.3 数据安全

- **传输加密**：使用 HTTPS 协议加密数据传输
- **密码加密**：使用 BCrypt 算法加密存储用户密码
- **敏感数据脱敏**：日志中的敏感数据进行脱敏处理
- **SQL 注入防护**：使用参数化查询防止 SQL 注入
- **XSS 防护**：对用户输入进行过滤和转义

#### 4.3.4 操作审计

- **操作日志**：记录用户的关键操作（登录、创建、修改、删除等）
- **日志内容**：操作人、操作时间、操作类型、操作对象、操作结果
- **日志存储**：日志数据独立存储，支持查询和导出
- **日志保留**：日志数据保留至少 1 年
