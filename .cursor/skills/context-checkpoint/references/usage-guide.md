# Context Checkpoint 使用指南

## 快速开始

当你在长对话中需要保存进度时，只需说：
- "创建检查点"
- "保存进度"
- "对话太长了，帮我创建个检查点"

Claude 会自动：
1. 分析当前对话的关键信息
2. 生成检查点文档
3. 提供恢复指令

## 检查点存储位置

所有检查点保存在：`.claude/checkpoints/`

文件命名格式：`checkpoint-YYYY-MM-DD-HH-MM.md`

## 如何恢复工作

在新对话中，粘贴恢复指令：

```
我正在继续之前的工作。请阅读检查点文档：
#file:.claude/checkpoints/checkpoint-2026-03-15-16-30.md

当前需要继续：[具体任务描述]
```

## 最佳实践

### 何时创建检查点

✅ **推荐时机**：
- 完成一个功能模块后
- 对话超过 50 轮
- 需要暂停工作，稍后继续
- 项目进入新阶段

❌ **不推荐时机**：
- 刚开始对话
- 任务还没有实质性进展
- 频繁创建（浪费存储空间）

### 检查点内容建议

**应该包含**：
- 关键决策和原因
- 已完成的重要工作
- 待办事项清单
- 技术实现细节
- 已知问题和注意事项

**不应该包含**：
- 敏感信息（密码、密钥）
- 过于详细的代码片段（可以引用文件路径）
- 临时性的调试信息

## 维护检查点

### 定期清理

建议每月清理一次旧的检查点文档：

```bash
# 查看所有检查点
ls -lh .claude/checkpoints/

# 删除旧的检查点
rm .claude/checkpoints/checkpoint-2026-01-*.md
```

### Git 配置

建议在 `.gitignore` 中添加：

```
.claude/checkpoints/
```

这样检查点文档不会被提交到版本控制系统。

## 常见问题

**Q: 检查点会自动创建吗？**
A: 不会。需要用户主动请求创建。

**Q: 可以创建多个检查点吗？**
A: 可以。每个检查点都是独立的文件。

**Q: 检查点文档可以手动编辑吗？**
A: 可以。检查点是普通的 Markdown 文件，可以随时编辑。

**Q: 如何在新对话中快速恢复？**
A: 使用 `#file:` 引用检查点文档，Claude 会自动读取并理解上下文。
